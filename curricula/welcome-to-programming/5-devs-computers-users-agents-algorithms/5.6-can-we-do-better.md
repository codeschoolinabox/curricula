# 5.6: Can We Do Better? (Qualitative Bridge to Chapter 5)

**Smallest problem (meta-level):** What's the smallest input where two decompositions produce DIFFERENT amounts of work? For palindrome: at length 3, the nested approach does 3 comparisons but two-pointer does 1. This is where the smallest-case ritual reveals algorithmic differences, not just edge cases.

**Growth pattern:** restructuring the decomposition — "can the same problem be decomposed differently to reduce total work?" This is a meta-level question: the object of study is the growth pattern itself.

This is a qualitative teaser, not a formal complexity analysis subchapter. The core skill: learners OBSERVE that different decompositions of the same problem produce different amounts of work, and articulate WHY in terms of growth patterns. The formal tools for measuring and comparing complexity (step-counting, growth rate tables, Big O notation) belong in Chapter 5. Here, the goal is to CREATE THE QUESTION that Chapter 5 will answer with rigor.

## Exercises and Themes

- **Palindrome three ways** (capstone comparison): build reversed string (4.1 — independent subproblems), nested pair comparison (4.4 — nested subproblems), two-pointer (4.5 — converging subproblems). Same behavior, three decompositions. Learners run all three with the same inputs, observe which "feels like more work," and articulate the difference in terms of growth patterns. All representation passes applied to all three — the richest comparison moment in the curriculum.

- **"All characters unique" revisited**: The nested decomposition compares every pair. What sub-problems get re-solved? What data structure would you NEED to avoid re-solving them? (Answer: a set or a frequency table — foreshadowing the next curriculum module.) Learners can't optimize it with current constraints, but they can _reason about what decomposition is missing_.

- **Substring search revisited**: Naive nested search from 4.4 — what happens on a mismatch? The algorithm restarts from the next position, throwing away everything it learned about the partial match. What if a mismatch told you something useful? This introduces the idea that algorithms can be designed by asking "what information am I currently throwing away?"

## Qualitative Gradient (internal sequencing)

1. **Observe**: Run the same problem with different decompositions. Which feels faster? Where does the extra work happen?
2. **Articulate**: Explain in terms of growth patterns WHY one decomposition does more work — what redundant sub-problems does it create?
3. **Imagine**: What would a better decomposition need? What data structures or approaches would enable it? (design thinking — even without implementing or formally measuring)

## Skills and Objectives

- 🥚 Running the same problem with different decompositions and observing which produces more steps
- 🥚 Identifying where redundant sub-problems cause extra work in a nested decomposition
- 🐣 Articulating WHY one decomposition does more work than another, using the growth pattern vocabulary from 4.1–4.5
- 🐣 Tracing palindrome three ways (4.1 reversed-string, 4.4 nested pairs, 4.5 two-pointer) and comparing the amount of work qualitatively — the capstone comparison moment of Chapter 4
- 🐥 Imagining what data structures or approaches would enable a better decomposition for "all unique" or substring search — reasoning about what's missing without implementing it
- 🐥 Recognizing that "feels like more work" is a genuine observation that deserves a formal answer — connecting the qualitative question to what Chapter 5 will answer

**Why this deserves its own subchapter:** "Can we do better?" is a qualitative shift in the _kind_ of question being asked. It's the transition from understanding decompositions to reasoning about _choosing between_ them. As a teaser, it creates motivation for Chapter 5's formal complexity analysis. As a bridge, it connects Chapter 4's growth pattern vocabulary to Chapter 5's measurement tools.

**Relationship to Chapter 5:** 4.6 creates the questions; Chapter 5 provides the formal tools to answer them. Learners leave 4.6 with intuitions ("the nested version does way more work") that Chapter 5 will sharpen into precise measurements (step-counting tables, growth rate comparisons, Big O notation).
