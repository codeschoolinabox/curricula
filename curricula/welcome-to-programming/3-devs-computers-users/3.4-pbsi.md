# 3.4: Purpose, Behavior, Strategy, Implementation

**A major milestone.** This subchapter introduces a 4-level framework for understanding programs that becomes the vocabulary for the rest of the curriculum and the bridge to algorithmic thinking.

## Language Features

- `break` and `continue`

## Skills and Objectives

- 🥚 **PBSI Framework**:
  - **Purpose**: Why a program exists, what it does for whom (the outer circle of the rhetorical diagram — for these programs the answer will always include "for learning \_\_\_")
  - **Behavior**: User-facing behavior without seeing the code (what a user experiences)
  - **Strategy**: The approach to solving the problem (the algorithm, in informal terms)
  - **Implementation**: The specific lines of code used

- 🥚 Three input validation strategies (and exploring their tradeoffs):
  1. All logic in the `while` head
  2. Boolean flag variable
  3. `while (true)` with `break`/`continue`

- 🥚 Exploring BSI (Behavior/Strategy/Implementation) variations in program logic — same behavior can be achieved through different strategies, each with different implementations

- 🥚 "Why" comments are about **strategy or behavioral correlations**, not implementation. A good comment explains _why this approach was chosen_ or _how this line connects to the user's experience_, not _what code is written on this line_.

- 🐣 **Describing programs** — the structured methodology for close reading at multiple zoom levels: zooming out (user's journey), zooming in (line-by-line), finding connections, labeling goals. See [`./describing-programs.md`](./describing-programs.md).

- 🐣 **Naming variables** — a structured approach: variable analysis (declaration, reads, assignments, types, scopes, checks) → generic names (role-based) → specific names (domain-based) → variable roles (fixed value, stepper, flag, gatherer, holder, temporary). See [`./naming-variables.md`](./naming-variables.md).

## Exercise Types

- All prior types
- Describing programs (full methodology: zoom out → zoom in → connections → goals)
- Naming variables (analysis template → generic/specific naming → role identification)
- Refactoring between the three input validation strategies
- Comparing BSI variations of the same program
