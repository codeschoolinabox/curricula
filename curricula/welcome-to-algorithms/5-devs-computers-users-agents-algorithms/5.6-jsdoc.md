# 5.6: JSDoc — Describing the Contract

Before writing a function, describe what it does. JSDoc is a formalized way to express the contract a function promises to fulfill: what it takes in, what it gives back, and what it does along the way.

**Intention before implementation.** This mirrors a pattern learners have practiced since Chapter 1: comments before code, structured block comments describing programs in Chapter 3, prose strategy descriptions before pseudocode in 5.1–5.5. JSDoc is the same idea, applied to functions: describe the _what_ before building the _how_.

## Why JSDoc First?

Functions are introduced in 5.7. But before writing functions, learners practice _describing_ them — because description is a comprehension skill, and comprehension comes before production.

Learners already understand the algorithms from 5.1–5.5. They can articulate what each algorithm takes in (a string), what it produces (a count, a boolean, a transformed string), and what strategy it uses. JSDoc gives them a structured notation for expressing this understanding:

```js
/**
 * Count the number of vowels in a string.
 *
 * @param {string} text - The string to scan for vowels.
 * @returns {number} The count of vowel characters (a, e, i, o, u).
 */
```

This is the _contract_ — what 5.7's function will promise to do. Writing the contract first forces clarity about purpose before getting tangled in implementation.

## The Program Format Connection

In Chapter 3, learners described whole programs using a structured format: input → process → output. JSDoc applies the same structure at a finer grain:

- `@param` = input (what the function receives)
- `@returns` = output (what the function gives back)
- The description = process (what strategy the function uses)

The difference: a program's I/O is user-facing (prompt/alert). A function's I/O is code-facing (parameters/return values). Same structure, different audience. This is the beginning of thinking about code as communication between developers, not just between developer and user.

## What Learners Practice

### Reading JSDoc

Given a complete JSDoc comment, learners answer:

- What does this function take in? What types?
- What does it return? What type?
- What does it do? Can you restate the description in your own words?
- What edge cases does the description suggest? (empty string? single character?)

### Writing JSDoc for Known Algorithms

Learners write JSDoc for algorithms they already understand from 5.1–5.5:

- Count vowels (5.1) — straightforward: string in, number out
- Longest run (5.2) — what exactly does it return? The length? The character? Both?
- Is palindrome (5.4/5.5) — string in, boolean out. How does the description hint at strategy?
- All characters unique (5.4) — same signature as palindrome, very different strategy

The exercise: write JSDoc _before looking at the code_, from memory of what the algorithm does. Then compare with the actual implementation — does the JSDoc accurately describe it? Would someone reading only the JSDoc know what to expect?

### Evaluating JSDoc Quality

Comparing good and bad JSDoc for the same function:

```js
// Too vague:
/** Does something with a string. */

// Too implementation-specific:
/** Loops through each character and increments a counter when it's a vowel. */

// Right level — describes the contract:
/** Count the number of vowels (a, e, i, o, u) in a string. */
```

The contract describes _what_, not _how_. A different implementation (regex, different loop structure) should satisfy the same JSDoc. This connects to 5.9's paradigm comparison: the same JSDoc could describe both an iterative and a regex solution.

## JSDoc Elements Covered

- `@param {type} name - description` — what the function receives
- `@returns {type} description` — what the function gives back
- Description block — what the function does, at the contract level
- Types used: `{string}`, `{number}`, `{boolean}` — the primitive types from the curriculum

**Not covered:** `@throws`, `@example`, `@typedef`, `@callback`, or any advanced JSDoc features. Just enough to describe the pure string functions learners will write in 5.7.

## Skills and Objectives

- 🥚 Reading a JSDoc comment and explaining what the function takes in, returns, and does
- 🥚 Writing JSDoc for an algorithm already understood from 5.1–5.5, using `@param`, `@returns`, and a description
- 🐣 Evaluating whether a JSDoc description is at the right level of abstraction — contract (what) vs. implementation (how)
- 🐣 Connecting JSDoc's structure to the program format from Chapter 3: `@param` = input, `@returns` = output, description = process
- 🐥 Recognizing that the same JSDoc could be satisfied by different implementations — the contract is implementation-independent

**Language notes:**

- JS: JSDoc is the standard documentation format, supported by editors and tooling
- Python: the equivalent is docstrings with type hints. The same pedagogical points apply, using `"""..."""` syntax and `:param`/`:returns:` conventions
