# 5.9: Regex — A Different Paradigm

Don't decompose at all — declare the pattern. Where 5.1–5.5 explored increasingly sophisticated ways to decompose problems into iterative sub-problems, regex asks: what if you described _what you're looking for_ instead of _how to look for it_? This is a fundamentally different relationship to sub-problems — the decomposition is implicit in the pattern engine, not explicit in your code.

**In the spirit of 3.6 (Plaintext Programs):** A fun interlude that reframes familiar problems through a completely different paradigm. Where 3.6 stripped away tools to see code as just text, 5.9 introduces a declarative alternative to see that the same behaviors can be expressed without iteration at all.

**Now with functions and tests:** With 5.6–5.8's function-wrapping and unit testing tools, regex solutions can be wrapped in named, documented, tested functions — making the comparison with iterative solutions even more concrete. Write a `countVowels` function iteratively (5.1-style) and declaratively (regex), give them the same JSDoc and tests, and compare what's inside.

**"Just enough" regex:** character classes, quantifiers, anchors, groups — enough to express the patterns learners have been implementing iteratively. NOT a comprehensive regex curriculum.

## The Pedagogical Point

Regex is a declarative pattern language. Learners have spent 5.1–5.5 building iterative solutions to string problems — counting, searching, matching, replacing. Regex does many of the same things in a single expression. The comparison is the exercise:

- "Count vowels" (5.1 iterative) vs. `/[aeiou]/g` — what does the regex express that the loop makes explicit?
- "Check if substring exists" (5.4 nested loop) vs. `.includes()` or a regex test — what work is being hidden?
- "Replace character X with Y" (5.1 iterative) vs. `.replace(/x/g, 'y')` — same behavior, different abstraction level
- "Find longest run" (5.2 state-dependent) vs. regex with backreferences — can regex even express this? Where does it struggle?

**What this reveals about algorithms:** Not every problem needs an algorithm. Some string operations are better expressed as patterns. But regex has limits — it can't express everything the iterative approach can (try expressing "longest run" or "palindrome check" purely in regex). Understanding where declarative breaks down reinforces why algorithmic thinking matters.

**Language notes:**

- JS: regex is built in (`/pattern/flags`, `.match()`, `.replace()`, `.test()`)
- Python: requires `import re` (`re.findall()`, `re.sub()`, `re.search()`)

## Skills and Objectives

- 🥚 Reading and explaining basic regex patterns using character classes, quantifiers, anchors, and groups
- 🥚 Comparing an iterative algorithm (from 5.1–5.5) with its regex equivalent and identifying what the regex expresses implicitly that the loop makes explicit
- 🐣 Identifying where regex can't express what an iterative approach can (longest run, palindrome check) and explaining why in terms of what state/history the algorithm needs
- 🐣 Recognizing regex as a declarative paradigm: describing WHAT to find rather than HOW to find it
- 🐥 Appreciating that "not every problem needs an algorithm" — some string operations are better expressed as patterns, and knowing which is which is itself a skill

**Explicit framing:** This is NOT expanding the core language feature set. Regex is presented as a comparison tool and a preview of declarative thinking, not a required skill. Like 3.6, the challenge is the _perspective shift_, not the code complexity.
