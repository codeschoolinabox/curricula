# 5.7: Functions — Naming, Wrapping, and Verifying Algorithms

Let's NAME, WRAP, and TEST the algorithms from 5.1–5.5. Functions give algorithms an identity: a name that communicates purpose (Chapter 1 callback), parameters that define input, and a return value that defines output. The JSDoc from 5.6 becomes the contract that the function fulfills.

## The Bridge

Learners have been writing algorithms as complete programs — with `prompt` for input, `console.log` for output, and the algorithm logic in between. Now we extract the algorithm logic into a **function**: a named, reusable piece of code with its own inputs (parameters) and output (return value).

This is not a new concept. It's the same algorithm, in a new wrapper. The wrapper makes it:

- **Nameable** — `countVowels` communicates what it does (Ch1: names as developer communication)
- **Testable** — call it with known inputs, check the output (formalizes the predict-execute-compare cycle)
- **Documentable** — the JSDoc from 5.6 describes the contract the function must satisfy
- **Reusable** — same algorithm, different inputs, no copy-paste

## Function Syntax

Arrow function expressions with `const`:

```js
/**
 * Count the number of vowels (a, e, i, o, u) in a string.
 *
 * @param {string} text - The string to scan for vowels.
 * @returns {number} The count of vowel characters.
 */
const countVowels = (text) => {
  let count = 0;
  for (const char of text) {
    if ('aeiou'.includes(char.toLowerCase())) {
      count = count + 1;
    }
  }
  return count;
};
```

This is a 5.1 algorithm — each element processed independently — wrapped in a function. The JSDoc from 5.6 sits above it. The `return` statement replaces `console.log`.

### Parameters and Return Values

- **Parameters** are the function's inputs. They replace `prompt()` — instead of asking the user, the function receives values directly.
- **Return values** are the function's output. They replace `console.log` — instead of printing to the console, the function hands a value back to the caller.
- A function that takes a string and returns a number/boolean/string is a _pure function_: its output depends only on its input, with no side effects. Every function in this subchapter is pure.

### Calling Functions

```js
const result = countVowels('hello world');
console.log(result); // 3
```

The caller provides the input. The function does the work. The caller receives the output. This separation is the point: the algorithm is now independent of how it gets its input and what happens with its output.

## Inline Assertions

Before formal unit testing (5.8), learners verify functions with inline assertions:

```js
console.assert(countVowels('hello') === 2, 'hello has 2 vowels');
console.assert(countVowels('') === 0, 'empty string has 0 vowels');
console.assert(countVowels('xyz') === 0, 'no vowels returns 0');
console.assert(countVowels('aeiou') === 5, 'all vowels');
```

`console.assert` is silent when the assertion passes. It logs an error when it fails. This is the simplest form of automated verification — each assertion is a test case from the predict-execute-compare cycle, made executable.

**Python equivalent:** `assert countVowels('hello') == 2, 'hello has 2 vowels'`

## Variable Scope

Functions create a new scope. Variables declared inside a function are local — they exist only while the function runs, then disappear. This is the first time learners encounter code that creates and destroys variables dynamically.

### Local Variables

```js
const countVowels = (text) => {
  let count = 0; // local — exists only inside this call
  for (const char of text) {
    // char is also local to each iteration
    if ('aeiou'.includes(char.toLowerCase())) {
      count = count + 1;
    }
  }
  return count; // the value escapes, but the variable doesn't
};
// count and char don't exist here
```

### Parameters Are Local

```js
const isPalindrome = (text) => {
  // text is local — it's a copy of whatever was passed in
  // changing text here doesn't change anything outside
};
```

### Scope Exercises

Scope is practiced through tracing, not lectures. Key exercises:

- **What can you see from here?** Given a function with local variables, identify which variables are accessible at different points in the code.
- **Parameter shadowing:** A function parameter has the same name as an outer variable. Which one does the function use? Trace to verify.
- **Return vs. log:** A function that `console.log`s internally vs. one that `return`s — what's the difference for the caller? (Log is a side effect; return is the output.)
- **Multiple calls:** Call the same function twice with different arguments. Are the local variables shared between calls? (No — each call creates fresh locals.)

## Exercise Progression

Following the PRIMMy scaffold from the chapter framework:

1. **Study** — read a complete function with JSDoc and assertions. Trace it. Does the function satisfy its JSDoc? Do all assertions pass?
2. **Fill in blanks** — given JSDoc + assertions + a function skeleton with blanks, fill in the missing logic
3. **Fix bugs** — given JSDoc + assertions + a buggy function, find and fix the bug. The assertions tell you what's wrong.
4. **Write the function** — given only the JSDoc and assertions, write the function from scratch
5. **Write everything** — given a prose description of what the function should do, write JSDoc, function, and assertions

At each level, scope tracing is integrated: learners identify local variables, trace parameter values, and verify that the function doesn't depend on or modify anything outside itself.

## Wrapping Known Algorithms

The primary exercise pattern: take an algorithm from 5.1–5.5 that learners already understand as a complete program, and restructure it as a function:

- Extract the algorithm logic from the program
- Define parameters (replacing `prompt`)
- Add a `return` statement (replacing `console.log`)
- Attach the JSDoc from 5.6
- Write assertions using test cases from the original program's doc comment

This is NOT new algorithmic thinking. It's restructuring known code into a new form — a comprehension exercise about the function abstraction itself.

## Skills and Objectives

- 🥚 Reading a function definition and explaining its parameters, return value, and what it does
- 🥚 Calling a function with arguments and using the return value
- 🥚 Wrapping a known algorithm (from 5.1–5.5) as a named function with parameters and a return value
- 🐣 Writing `console.assert` statements to verify a function's behavior against known inputs and expected outputs
- 🐣 Tracing variable scope: identifying which variables are local to a function, which are parameters, and which are accessible from outside
- 🐣 Explaining the difference between `return` (function output) and `console.log` (side effect)
- 🐥 Writing a complete function from a JSDoc contract: reading the JSDoc, implementing the logic, and verifying with assertions
- 🐥 Recognizing that the function's local scope isolates the algorithm — each call starts fresh, and nothing leaks out

**Language notes:**

- JS: arrow function expressions (`const name = (params) => { ... }`)
- Python: `def name(params):` with `return`. Same scope concepts, slightly different syntax.
