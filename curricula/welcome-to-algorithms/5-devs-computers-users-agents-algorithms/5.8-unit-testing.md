# 5.8: Unit Testing — Formalizing Verification

Inline assertions (5.7) work, but they don't scale. When a function has many test cases, you need structure: grouping related tests, naming what each test checks, and getting clear reports of what passed and what failed. Unit testing frameworks provide this structure.

## From Assertions to Tests

In 5.7, learners wrote:

```js
console.assert(countVowels('hello') === 2, 'hello has 2 vowels');
console.assert(countVowels('') === 0, 'empty string has 0 vowels');
```

These assertions are scattered. They're silent when they pass. If one fails, it's hard to tell which one. Unit testing restructures the same checks into organized, readable, reportable tests:

```js
describe('countVowels', () => {
  it('should count vowels in a normal string', () => {
    expect(countVowels('hello')).toEqual(2);
  });
  it('should return 0 for an empty string', () => {
    expect(countVowels('')).toEqual(0);
  });
  it('should return 0 when there are no vowels', () => {
    expect(countVowels('xyz')).toEqual(0);
  });
  it('should count all vowels', () => {
    expect(countVowels('aeiou')).toEqual(5);
  });
});
```

### The Three Parts

- **`describe`** — groups tests for a single function. The string matches the function name.
- **`it`** — one test case. The string says what behavior this test checks. Read it as: "it should [do something]."
- **`expect(...).toEqual(...)`** — the actual check. Same logic as `console.assert`, but with clear output: which value was expected, which was received.

### Single-File Pattern

All tests live in the same file as the function, below the function definition. No imports, no modules, no separate test files. One file = one function + its JSDoc + its tests.

```js
// --- function ---

/**
 * Check if a string is a palindrome.
 * @param {string} text - The string to check.
 * @returns {boolean} True if the string reads the same forwards and backwards.
 */
const isPalindrome = (text) => {
  // ... implementation ...
};

// --- tests ---

describe('isPalindrome', () => {
  it('should return true for a palindrome', () => {
    expect(isPalindrome('racecar')).toEqual(true);
  });
  it('should return true for a single character', () => {
    expect(isPalindrome('a')).toEqual(true);
  });
  it('should return true for an empty string', () => {
    expect(isPalindrome('')).toEqual(true);
  });
  it('should return false for a non-palindrome', () => {
    expect(isPalindrome('hello')).toEqual(false);
  });
});
```

## Progressive Test Writing

Test exercises follow a scaffold from comprehension to production:

### 1. Study: Read and Run Tests

Given a complete file (function + tests), learners:
- Read each `it` block and predict whether it will pass or fail
- Run the tests and compare predictions to results
- Explain _why_ each test passes or fails in terms of the function's logic

### 2. Write Tests for a Working Function

Given a function with JSDoc but no tests, learners write tests:
- Start from the JSDoc: what does `@param` suggest about inputs to test? What does `@returns` suggest about expected outputs?
- Use the test case sequencing from the chapter framework: happy path → edge case → subtle case
- The JSDoc contract tells you _what_ to test. The function tells you _what to expect_.

### 3. Pass Tests (Fix the Function)

Given tests + JSDoc but a buggy or incomplete function, learners fix the implementation until all tests pass. Variants:

- **Blanks** — function body has strategic blanks to fill in
- **Bugs** — function has specific errors that cause test failures
- **Empty** — function body is completely empty, write from scratch

This is test-driven development (TDD) in its simplest form: the tests define success, the learner writes code to achieve it.

### 4. Red-Green-Refactor

The full TDD cycle, applied to algorithms learners already understand:

1. **Red** — write a test for one behavior. Run it. It fails (red).
2. **Green** — write the minimum code to make that test pass. Run it. It passes (green).
3. **Refactor** — clean up the code without changing behavior. Run tests again. Still green.
4. Repeat for the next behavior.

This is a natural extension of the predict-execute-compare cycle: the test is the prediction, execution checks it, and the comparison tells you if your code is correct.

## Test Case Design

Designing test cases is itself a skill. The test case sequencing from the chapter framework applies:

1. **Happy path** — a straightforward input where the function should clearly work
2. **Edge cases** — empty string, single character, all same characters. What does the JSDoc imply about these?
3. **Boundary cases** — inputs that sit at the transition between behaviors (e.g., the smallest palindrome, a string with exactly one vowel)

The question "what inputs should I test?" connects directly to the analytical chain's "what is the smallest version of this problem?" from 5.0. The smallest cases ARE the first test cases.

## Connection to Chapter 6

Unit tests make functions comparable. In Chapter 6, learners will compare the _efficiency_ of different functions that satisfy the same JSDoc contract. Having named, tested functions means:

- The same tests verify that all implementations produce correct results
- The functions can be called with increasing input sizes to measure performance
- The comparison is apples-to-apples: same contract, same tests, different strategies

## Skills and Objectives

- 🥚 Reading a `describe`/`it`/`expect` test and explaining what behavior it checks
- 🥚 Running tests and interpreting the output: which passed, which failed, and why
- 🐣 Writing tests for a function based on its JSDoc contract, covering happy path, edge cases, and boundary cases
- 🐣 Fixing a buggy function by using test failures as diagnostic information — each failing test points to a specific behavior that's broken
- 🐣 Connecting test case design to the analytical chain: smallest cases (0, 1, 2 elements) map directly to edge case tests
- 🐥 Practicing the red-green-refactor cycle: write a failing test, make it pass, clean up, repeat
- 🐥 Designing test cases that would distinguish between different implementations of the same contract (e.g., tests that pass for the two-pointer palindrome but would also pass for the reversed-string palindrome)

**Language notes:**

- JS: uses a lightweight testing framework (e.g., Jest-style `describe`/`it`/`expect`) in a single file
- Python: uses `unittest` or `pytest` with equivalent patterns (`class TestX(unittest.TestCase)` / `def test_x():`)
