# 4.2: Agents and Developer Communication

Revisits Chapter 1 skills (comments, naming, formatting) through the collaboration lens. The simplest code context: programs as developer-to-developer communication.

## Skills and Objectives

- 🥚 Read LLM-generated comments and evaluate whether they are helpful for developers (using Ch 1 standards)
- 🥚 Read LLM-suggested variable names and evaluate whether they follow naming conventions and variable roles
- 🐣 **Perspective-Take:** Given LLM-generated comments or names, hypothesize what the LLM "saw" in the code that led to those choices
- 🐣 **Articulate:** Write prompts that give the LLM enough context to generate useful developer-facing output (comments, names, formatting)
- 🐣 **Drafting structured comments for programs that don't exist yet:** Write full user-story-based documentation in the Ch 3.3 format (Frogopedia-style) for an imaginary program, then have the LLM generate code from it. Or: describe a program idea, have the LLM generate the structured comments, then evaluate whether they capture your intent. Tests both articulation (writing for the LLM) and evaluation (reading LLM output against developer communication standards).
- 🐥 **Calibrate:** Where is the LLM good at developer-facing output? Where does it fail? (e.g., "what" vs. "why" comments)
- 🐥 **Delegate:** Is this a task where the LLM adds value (generating many name options quickly) or where you should do it yourself (understanding _why_ a name fits)?
- 🐔 **SOLO check-in:** Are you building structure (learning naming conventions) or leveraging structure (quickly generating names for a project)? Different goals → different collaboration approaches.

## Exercise Types

- Have LLM generate comments for existing code → critically evaluate the comments (developer audience)
- Have LLM suggest variable names → evaluate against naming conventions and variable roles from Ch 3.4
- Marking, tracing, and describing exercises from [`../-1-getting-started/exercise-types.md`](../-1-getting-started/exercise-types.md) — now applied to LLM output
- Frogopedia-style documentation exercises: write structured comments → generate code from them → evaluate the result
- Reverse exercise: give the LLM code → have it generate structured comments → evaluate whether the comments accurately describe the code
