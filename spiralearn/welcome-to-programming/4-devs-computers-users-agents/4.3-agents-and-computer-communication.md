# 4.3: Agents and Computer Communication

Revisits Chapter 2 skills (variables, state, tracing) through the collaboration lens. This is where a critical distinction emerges: analytical tools (debuggers, linters) vs. LLM-based code analysis.

## Skills and Objectives

- 🥚 Trace LLM-generated code using predictive stepping (Ch 2 skills applied to a new source)
- 🐣 Have the LLM trace code, then evaluate whether its traces track state correctly
- 🐣 Have the LLM explain code, then describe whether the explanation matches actual execution
- 🐣 **Perspective-Take:** LLMs often produce plausible-looking but wrong traces. Why? (Pattern matching vs. execution)
- 🐥 **Calibrate:** LLMs are better at _generating_ code than _tracing_ it accurately. Why does this matter?
- 🐥 Evaluate LLM-generated traces for correctness
- 🐥 **Delegate:** When should you trace yourself vs. ask the LLM to trace? (Answer: if you can't trace it yourself, you can't evaluate the LLM's trace.)
- 🐔 **SOLO check-in:** Tracing is a foundation-building skill. If you can't trace code yourself, you can't evaluate an LLM's trace. This is why Human-Led is the right approach here — you need the skill before you can delegate it.

## Analytical vs. Heuristic Program Analysis

A key distinction for this subchapter. Two fundamentally different ways to analyze code:

**Analytical tools** (debuggers, linters, type checkers, static analysis) perform _formal, provable_ analysis. They parse abstract syntax trees, traverse call graphs, track data flow. They are always correct within their scope.

**LLMs** do something fundamentally different. They pattern-match, approximate, and sometimes hallucinate. Su & McMillan (2025, "Do Code LLMs Do Static Analysis?") proved that LLMs are NOT implicitly performing static analysis — they use entirely different mechanisms.

| Dimension                | Analytical Tools (debugger, linter)      | LLM Code Analysis                                    |
| ------------------------ | ---------------------------------------- | ----------------------------------------------------- |
| Mechanism                | Formal parsing, provable traversal       | Pattern matching, statistical correlation              |
| Correctness              | Sound (within scope)                     | Approximate, hallucination-prone                       |
| Obfuscated code          | Handles correctly                        | Performance degrades significantly                     |
| High-level comprehension | Struggles (no "understanding")           | Stronger at holistic/natural language descriptions     |
| Setup                    | Requires tooling per language            | Works across languages heuristically                   |

**The pedagogical point:** Both have strengths the other lacks. Analytical tools are provably correct but can't tell you "what this program is for." LLMs can give you high-level comprehension but may be wrong about specifics. Knowing which to use when is a collaboration skill.

## Exercise Types

- Have LLM generate simple programs → trace the generated code yourself (predictive stepping)
- Have LLM trace code → evaluate its traces against your own (where does it go wrong?)
- Have LLM explain code → compare the explanation to actual execution (using the debugger)
- Side-by-side exercises: analyze the same program with a debugger AND an LLM. Compare what each reveals. What does the debugger tell you that the LLM misses? What does the LLM tell you that the debugger can't?
- Connection to logging, state tracking, and predictive stepping from Ch 2
