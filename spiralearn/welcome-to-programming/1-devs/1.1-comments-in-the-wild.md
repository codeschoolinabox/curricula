# 1.1: Comments in the Wild

A fun interlude. Here is a curated collection of real comments found in real
codebases by real developers. Some are helpful, some are funny, some are
desperate, some are poetry. All of them are one developer talking to another
developer — which is exactly what you've been practicing.

Some of these include code you haven't learned yet — that's fine. Focus on the
comments. Think about: Who were they writing to? What were they trying to
communicate? Did it work?

All comments are sourced from
[this Stack Overflow discussion](https://stackoverflow.com/questions/184618/what-is-the-best-comment-in-source-code-you-have-ever-encountered)
and other documented origins so you can verify they're real.

## Comments that warn the next developer

> ```
> // Magic. Do not touch.
> ```
>
> — [source](https://stackoverflow.com/a/185106)

> ```
> //
> // Dear maintainer:
> //
> // Once you are done trying to 'optimize' this routine,
> // and have realized what a terrible mistake that was,
> // please increment the following counter as a warning
> // to the next guy:
>
> total_hours_wasted_here = 42
>
> ```
>
> — [source](https://stackoverflow.com/a/482129)

> ```
> /*
>  * You may think you know what the following code does.
>  * But you dont. Trust me.
>  * Fiddle with it, and youll spend many a sleepless
>  * night cursing the moment you thought youd be clever
>  * enough to "optimize" the code below.
>  * Now close this file and go play with something else.
>  */
> ```
>
> — [source](https://stackoverflow.com/a/189859)

> ```
> /* You are not expected to understand this */
> ```
>
> — from the
> [Unix V6 operating system source code](https://thenewstack.io/not-expected-understand-explainer/), 1975.
> The authors later admitted the code was actually buggy and they didn't fully
> understand it either.

## Comments that are brutally honest

> ```
> //When I wrote this, only God and I understood what I was doing
> //Now, God only knows
> ```
>
> — [source](https://stackoverflow.com/a/316233)

> ```
> // drunk, fix later
> ```
>
> — [source](https://stackoverflow.com/a/185181). The person who posted this
> added: "Wish I were kidding. And knowing the developer who wrote the code, I
> think he meant it literally."

## Comments that are poetry

> ```
> // Replaces with spaces the braces in cases
> // where braces in places cause stasis
> ```
>
> — [source](https://stackoverflow.com/a/766363)

> ```
> long long ago; /* in a galaxy far far away */
> ```
>
> — [source](https://stackoverflow.com/a/615845). `long long` is a data type in
> C, and `ago` is a valid variable name — so this is simultaneously valid code
> and a Star Wars quote.

> ```
> stop(); // Hammertime!
> ```
>
> — [source](https://stackoverflow.com/a/186309)

## Skills and Objectives

- 🥚 Reading real developer-to-developer communication from production code
- 🥚 Recognizing that comments serve many purposes: explaining why, warning
  future developers, expressing frustration, documenting intent, apologizing for
  hacks
- 🐣 Evaluating which comments communicate effectively and which don't — and why
- 🐣 Noticing that the best comments don't describe _what_ the code does, but
  _why_ it does it (or why you shouldn't touch it)

## Design Notes

- The comments are curated for variety and for humor that works without
  understanding the surrounding code. Ch1 learners don't know code yet, so the
  comments must stand on their own as human communication.
- Every comment links to a verifiable source. This collection is sourced
  primarily from
  [this Stack Overflow discussion](https://stackoverflow.com/questions/184618/what-is-the-best-comment-in-source-code-you-have-ever-encountered)
  and the
  [Unix V6 history](https://thenewstack.io/not-expected-understand-explainer/).
- This is a palate cleanser and a celebration of the human side of code before
  Chapter 2 introduces the computer as audience. After this, comments stop being
  the whole story — but they never stop being part of it.
- The "useless comment" pattern (describing exactly what the code already says)
  is deliberately absent from this curated list, because understanding _why_
  `i++; // increment i` is funny requires knowing what `i++` means. But it
  plants a seed to revisit once learners can read code.
