# 3.2: Variable Program Behaviors

Programs can now produce different outputs based on the input. This is where
programs become genuinely interesting to study — the same code can follow
different paths.

## Language Features

- Conditionals: `if` / `else` / `else if` (JS) / `elif` (Python)
- Explicitly excluded: `switch`/`case` — never

## Skills and Objectives

- 🥚 Input/output pairs (test cases) in top-level doc comment
- 🥚 Test coverage: are all conditional paths covered by your test cases?
- 🐣 **Fixing bugs** — distinct from fixing errors (Chapter 2.1). Bugs are when
  code runs without errors but produces wrong behavior. The structured approach:
  what _should_ it do? → what _does_ it do? → how _does_ it work? → which
  line(s) are surprising? → experiment. See
  [`./fixing-bugs.md`](./fixing-bugs.md).
- 🐣 **Modifying programs** — making small, tracked changes to working programs.
  One change at a time, predict the effect, run, note the result. This builds
  toward writing programs by first practicing controlled modification. See
  [`./modifying-programs.md`](./modifying-programs.md).

## Exercise Types

- All prior types
- Fixing bugs (structured experiment template)
- Modifying programs (challenge-based modifications with tracked experiments)
- Comparing programs (same behavior, different conditional structures)
