# 3.3: Validating User Input

Programs can now ensure that user input meets requirements before proceeding.
This introduces loops and establishes a program structure pattern that will be
used consistently: **input + validation → logic → output**.

## Language Features

- `while` loops
- Type coercion (in the context of validating input types)

## Skills and Objectives

- 🥚 **Program structure pattern**: input + validation (while loop), then logic
  (conditional), then output. This pattern is followed consistently for clarity,
  for algorithm isolation later, and to implicitly prepare for functions.
  - A `while` loop to validate a basic condition
  - Followed by a conditional to process the input
  - Followed by the final output
- 🥚 Getting numbers from users: casting to number, validating that the cast
  succeeded, validating the range
- 🥚 Full user-story-based top-level comments, for example:

  ```js
  /* Frogopedia
  
    <plain english description of program behavior>
  
    A user can share cool facts about frogs.
    - given the user cancels, they will be prompted again
    - given the input does not contain "frog", the user is told so and prompted again
    - given the input contains "frog" (case-insensitive), the program learns something
  
    test cases:
    - given the user cancels → (no output, input is invalid)
    - given "hello" → (no output, input is invalid)
    - given "frog" → 'i just learned something cool about frogs!\n\n- "frog"'
    - given "FROGZZ" → 'i just learned something cool about frogs!\n\n- "FROGZZ"'
  */
  ```

- 🐣 **Logging strategies** — structured approaches to `console.log` placement:
  logging program structure (beginning/end/chunks), logging variables (name,
  type, value), logging control flow (which path was taken, loop iterations).
  The golden rule: logs should be windows into your program, not part of the
  program's logic. See [`./logging-state.md`](./logging-state.md).
- 🐣 **Backtracing** — tracing _backwards_ from a program's output to its input.
  Instead of "what happens next?", you ask "why did this happen?". See
  [`./backtracing.md`](./backtracing.md).

## Exercise Types

- All prior types
- Logging exercises (different strategies, finding the right balance)
- Backtracing exercises (working from output back to input)
- Tracing while loops (iteration count, exit conditions)
