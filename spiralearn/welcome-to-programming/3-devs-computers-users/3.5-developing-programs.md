# 3.5: Developing Programs

**Development is 90% modification.** The moment you write code, it's someone
else's — and most of the time you're adjusting your own, not writing new lines.
This subchapter builds up to writing programs from scratch by first practicing
modification and review, then moving to guided writing.

## Language Features

- (nothing new — this chapter is about skills, not features)

## Skills and Objectives

- 🥚 **Linting** — finding and fixing style issues
- 🐣 Exercises where you change a program's implementation or strategy without
  changing its behavior (building on the modification skills from 3.2). See
  [`./modifying-programs.md`](./modifying-programs.md).
  - Discussions of style and preference
  - Readability vs. efficiency (efficiency is not the priority at this stage)
- 🐣 **Refactoring** — a more formal framing of the same idea: changing code
  without changing behavior. See [`./refactoring.md`](./refactoring.md).
- 🐣 **Code review** — a structured review template covering behavior (title,
  description, user story, test cases), goals (names, progression, data),
  comments ("why" not "what", quantity/quality), linting, variables (location,
  initial values, roles, domain), logs, and logic (operations, control flow).
  See [`./code-review.md`](./code-review.md).
- 🐥 **Writing programs from spec** — graduated levels of scaffolding: (0)
  stepped examples showing the incremental process, (1) starter code with
  behavior and goals, (2) specs and goals without code, (3) only specs. The
  process: understand behavior → plan goals → English solution → pseudocode →
  write/predict/check/repeat → test → debug. See
  [`./from-spec.md`](./from-spec.md).
- 🐔 **Reverse engineering** — studying obfuscated programs and recreating their
  behavior: describe behavior → plan goals → plan strategy → write code. See
  [`./reverse-engineering.md`](./reverse-engineering.md).
- 🐔 Writing programs from unstructured guidance: plain English prompts, word
  problems, imagining your own

### Studying with LLMs: From Study Partner to Development Partner

This is the subchapter where LLMs shift from being purely study partners to also
being relevant as _development_ partners. Learners now write programs from spec,
review code, and refactor — all activities where LLMs can participate. The key
difference from earlier chapters: learners must critically evaluate LLM output
using the skills they've built (PBSI, variable roles, tracing, code review
checklists).

- LLMs can assist with spec interpretation, code review, and debugging
- LLMs can generate code from specs — but can the learner trace, describe, and
  review what was generated?
- The emphasis remains on comprehension: using AI to produce code you don't
  understand is not development, it's delegation

See the "Specs: LLM Study Strategy" in
[`../-1-getting-started/exercise-types.md`](../-1-getting-started/exercise-types.md).

## Exercise Types

- Refactoring exercises (change implementation, preserve behavior)
- Code review exercises (using the structured template, ideally with a study
  partner)
- From-spec exercises at all 4 scaffolding levels
- Reverse engineering exercises (mystery programs)
- Modifying programs (more complex challenges than Chapter 3.2)
