# 5.2: Adjacent Element Relationships

**Smallest problem:** Empty string → nothing to compare (0 transitions). One
character → no "previous" to compare to — reveals that the first iteration is
special and directly informs state initialization (what should `previousChar`
start as?). Two characters → the first real comparison, the minimal case where
the growth pattern is visible.

**Growth pattern:** n sequential, dependent steps — "for each character, answer
a question that depends on what you found for the previous character." The
answer at position k depends on the answer at position k-1. Adding one more
character extends the chain of dependency by one link.

**State:** gatherer + holder(s) for "what came before" — previous char, running
count, best-so-far. Follows from the dependency: to compare current with
previous, you must carry the previous forward.

**Scope of attention:** current character vs. previous character (or accumulated
local state) — follows from the growth pattern.

**Iteration follows as:** still single pass, but state now carries history
forward.

The qualitative jump: what the algorithm does with character k depends on what
it saw at character k-1. Each step is no longer self-contained — it depends on
the prior step's result.

## Algorithms

- Check if string is sorted/non-decreasing (track: previous char, flag)
- Count the number of runs (track: previous char, increment on change)
- Longest run of the same character (track: current char, run length, best
  length, best char)
- Run-length encoding — `"aaabbc"` → `"3a2b1c"` (track: current char, count;
  emit on change)

"Longest run" is the central algorithm — it combines rich state tracking (4
variables) with multi-outcome decisions (extend OR reset AND maybe update best),
while remaining intuitive ("what's the longest streak?"). RLE is the most
complex algorithm at this level, placed last in the decision gradient; it may
need motivating context (compression, data encoding) since learners may not know
what it is.

## Decision Complexity Gradient (internal sequencing)

1. **Simple state-dependent** — one comparison to previous: "check if sorted"
   (is current >= previous?), "count runs" (did the character change?)
2. **Multi-outcome state-dependent** — decision has multiple consequences:
   "longest run" (extend current run OR reset run AND maybe update best)
3. **Compound state-dependent** — multiple interacting conditions with emit
   logic: run-length encoding (current == previous? increment count. Different?
   emit count+char, reset count, update tracked char)

## Skills and Objectives

- 🥚 Identifying the dependent-chain growth pattern: each step's answer depends
  on the previous step's result
- 🥚 Recognizing that dependent-chain algorithms require holder variables to
  carry forward what the previous step found — connecting to Ch3's variable
  roles vocabulary
- 🥚 Tracing algorithms that maintain both a gatherer and one or more holder
  variables across iterations (e.g., "longest run" with 4 variables)
- 🐣 Explaining why the first iteration is special and how this informs state
  initialization — using the smallest-problem analysis (0, 1, 2 elements) to
  determine what `previousChar` should start as
- 🐣 Articulating the 5.1→5.2 comparison: "count vowels" vs. "count runs" — same
  counter pattern, same code shape, but fundamentally different growth patterns
  and different state requirements
- 🐣 Producing all 7 representations for state-dependent algorithms, including
  loop invariants that capture the dependency relationship (what's always true
  about `previous` relative to the loop index?)
- 🐥 Recognizing compound state-dependent decisions (like run-length encoding)
  where multiple interacting conditions produce emit-on-change logic

**Why these together:** They all have the same growth pattern: each step depends
on the previous step's result. The loop body's behavior _changes based on
history_. But the dependency is local — you only need to know what just
happened, not the entire traversal. All still single-pass.

**Key comparison moment (5.1→5.2):** "count vowels" vs. "count runs" — same
counter pattern, but the growth is fundamentally different. In 5.1, each step is
independent ("is this character a vowel?"). In 5.2, each step depends on the
previous ("did the character change?"). Same code shape, different growth
pattern — and the state requirements follow directly from that difference.
