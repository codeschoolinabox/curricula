# 5.4: Global Element Relationships

**Smallest problem:** One character → always unique, no substring to find. Two
characters → the minimum case where the growth pattern is visible (one
comparison for uniqueness, one alignment for substring search). Exposes edge
cases like "what if the pattern is longer than the text?" Note: without
arrays/sets, comparing every pair is the ONLY option for "all unique" — this
constraint-forced approach creates a teaching moment in 6.1 ("Can We Do
Better?"): "what data structure would you NEED?"

**Growth pattern:** nested subproblems — "for each position, spawn a new
sub-problem that examines other positions." The outer question ("is this
character unique?", "does a match start here?") can't be answered without
solving an inner subproblem that scans part or all of the string.

**State:** multiple interacting trackers — follows from nesting: the outer
subproblem's state must persist while the inner subproblem runs, and the inner
result feeds back into the outer.

**Scope of attention:** current element vs. any/all positions in the string —
follows from the nested structure: each inner subproblem needs to see positions
beyond the current one.

**Iteration follows as:** nested loops — forced by the nested subproblem
structure. The inner loop IS the inner subproblem.

The algorithm needs to relate positions that may be far apart. For each
position, it spawns a sub-problem that examines other positions. This is where
subproblem structure drives the iteration structure to nesting.

## Algorithms

- Check if all characters are unique (compare each pair)
- Check if a substring/pattern exists (slide pattern across text)
- Count substring occurrences
- Find first position of a substring

## Decision Complexity Gradient (internal sequencing)

1. **Simple nested comparison** — inner loop makes one comparison per iteration:
   "all characters unique" (for each char, check against all others — decision
   is just equality)
2. **Nested with early exit logic** — break out of inner loop on match: "check
   if substring exists" (match so far? keep comparing. Mismatch? break, advance
   outer)
3. **Nested with counting/tracking** — inner loop accumulates a result: "count
   substring occurrences", "find first position of substring" (inner loop
   decisions feed back into outer loop state)

## Skills and Objectives

- 🥚 Identifying the nested-subproblem growth pattern: each position spawns a
  new sub-problem that examines other positions
- 🥚 Tracing nested-loop algorithms where the inner loop IS the inner
  sub-problem — tracking state that persists across inner-loop iterations
- 🥚 Recognizing that nested subproblems require multiple interacting state
  variables (outer position, inner position, match tracking)
- 🐣 Articulating why nested algorithms do dramatically more work than
  single-pass algorithms — connecting the "feels like more work" observation to
  the nested decomposition structure
- 🐣 Articulating the 5.2→5.4 comparison: "longest run" (single pass, local
  dependency) vs. "all unique" (nested) — same question type, different
  decomposition, different amount of work
- 🐣 Producing all 7 representations for nested algorithms, including nested
  structure in flowcharts and inner/outer relationships in loop invariants
- 🐥 Identifying where early-exit logic (`break`) reduces actual work without
  changing the worst-case growth pattern — a first taste of best-case vs.
  worst-case reasoning

**Why these together:** They all have the same subproblem structure: an outer
problem that can only be solved by spawning inner sub-problems at each position.
The inner loop is a _consequence_ of the nested decomposition: when the
subproblem at position k requires examining positions k+1 through n, you must
iterate over them.

**Key comparison moment (5.2→5.4):** "longest run" (single pass, local
dependency) vs. "all unique" (nested subproblems) — both ask about character
relationships, but "longest run" decomposes into sequential dependent
subproblems (each only needs the previous answer), while "all unique" decomposes
into nested subproblems (each needs to check against ALL other positions). Same
question type, different decomposition, dramatically different amounts of work.
The first time learners encounter: "could this be done with less work?"

**Chapter 5 foreshadowing:** This is where the qualitative observation of "more
work" becomes most vivid. Learners can feel that nested algorithms do
dramatically more work than single-pass ones. The formal measurement
(step-counting, growth rate tables, Big O) belongs in Chapter 6, but the
intuition is built here. Chapter 6.1 ("Can We Do Better?") will make this
observation explicit.
