# 5.1: Each Element Independently

**Smallest problem:** Empty string → result is empty (or 0, or true). One
character → the algorithm's logic must work for one character in isolation. This
reveals that each element is processed independently.

**Growth pattern:** n independent, identical steps — "for each character, answer
the same question." The answer at position k doesn't depend on the answer at any
other position. Adding one more character to the input adds one more independent
step.

**State:** one gatherer (accumulator) — follows from independence: no need to
carry history, just the growing result.

**Scope of attention:** current character only (or vs. a fixed reference like
`"aeiou"`) — follows from independence.

**Iteration follows as:** single sequential pass.

First subchapter with full representation sequence. **This is where the
analytical chain is explicitly taught.** The first few algorithms (reverse,
count vowels) are deliberately simple so that the focus is on learning the chain
of reasoning and the representation formats, not on algorithmic complexity.
Learners practice: (1) trying the smallest cases first, (2) describing how the
solution grows, (3) identifying state, (4) connecting to language features. The
representations may be trivially simple at this level — this is intentional, as
learners practice the FORMAT before the CONTENT gets hard.

## Algorithms

- Reverse a string (no decision — same action every iteration)
- Mirror a string (reverse + append, or build from both sides)
- Palindrome check via reversed string (build reversed, compare with `===` —
  needed for 6.1 capstone comparison)
- Remove all instances of a character (simple filter: if match, skip)
- Replace character X with character Y (two-branch: if match, substitute; else
  keep)
- Count specific characters — vowels, digits, spaces (simple filter + numeric
  gatherer)
- Check if a character exists, with early `break` (introduces `break` as
  algorithmic tool)
- Find first/last position of a character (`break` on match vs. overwrite —
  strategy pair)
- Caesar cipher — shift each character by a fixed offset (`charCodeAt`/`ord` +
  `String.fromCharCode`/`chr`)
- Simple parenthesis/bracket balance checking (numeric gatherer that increments
  AND decrements — boundary case showing that accumulation can be non-monotonic)

## Decision Complexity Gradient (internal sequencing)

1. **No decision** — same action every iteration: reverse, mirror, Caesar cipher
2. **Simple filter** — if condition, act (else skip): count vowels, remove
   character, check if char exists (with `break`)
3. **Two-branch** — if/else with different actions: replace X with Y, find first
   vs. last position (break vs. overwrite), parenthesis balance (increment vs.
   decrement)

The accumulation pattern (`result = ""` + loop) is the constant; decision logic
is the variable that drives difficulty within this subchapter.

## Skills and Objectives

- 🥚 Applying the analytical chain to independent-subproblem algorithms:
  smallest problem → growth pattern → state → language features
- 🥚 Identifying the independent-subproblem growth pattern: each step is
  identical regardless of position, adding one element adds one independent step
- 🥚 Tracing and writing algorithms that accumulate a result
  character-by-character using a single gatherer variable (`let result = ""`)
- 🥚 Producing all 7 representations for an algorithm: prose, flowchart,
  pseudocode, decision table, loop invariant, state transition, pattern schema
- 🐣 Recognizing the decision complexity gradient within this growth pattern: no
  decision (reverse, mirror) → simple filter (count vowels, remove char) →
  two-branch (replace X with Y, find first/last position)
- 🐣 Performing adjacent and distant transitions between representations (e.g.,
  code → prose vs. code → loop invariant) and explaining why the transition is
  correct
- 🐣 Evaluating LLM-generated representations for precision and completeness —
  identifying where the LLM skips a step, is vague, or gets the invariant wrong
- 🐥 Classifying algorithms as instances of the filter-accumulate pattern schema
  and recognizing which 5.1 algorithms belong to the same family

**Why these together:** The growth pattern is independent — no iteration needs
to "remember" anything about previous steps. The decision at each character
depends only on the character itself (or its relationship to a fixed reference
like `"aeiou"` or `"("`). State tracks the growing output, not the traversal
history. Parenthesis balance is a boundary case: the counter accumulates across
iterations, but each iteration's decision (increment or decrement?) depends only
on the current character, not on the previous character. It's the richest
5.1-level algorithm — foreshadowing 5.2's state complexity without requiring
adjacent-element comparison.
