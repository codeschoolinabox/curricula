# 6.0: Counting Setup — The Methodology

Before analyzing complexity, learners need a tool and a method. This subchapter introduces both: how to count operations, how to organize the counts, and how to use the count table as the foundation for all subsequent analysis.

## The Core Question

"How many times does each operation execute?"

This is the first question in the analytical chain (5+1), and it's the foundation for everything in Chapter 6. Before learners can reason about growth, dominance, explanation, or classification, they need to be able to COUNT.

## What Counts as a "Step"?

Not everything. Learners choose what to count based on what they want to understand:

- **Comparisons** — how many times does the algorithm ask a question? (`if`, loop conditions, `===`)
- **Assignments** — how many times does a variable get a new value? (`count = count + 1`, `let result = ''`)
- **Iterations** — how many times does the loop body execute?
- **Function calls** — how many times is a function invoked? (relevant when comparing implementations in 6.1+)

The choice of what to count is itself a pedagogical decision. Start with iterations (easiest to observe), then refine to comparisons and assignments as learners develop intuition for which operations matter.

## The Count Table

The count table is the chapter's foundational representation. It maps input size to operation counts:

| Input | Input Size (n) | Iterations | Comparisons | Assignments |
| ----- | -------------- | ---------- | ----------- | ----------- |
| `"a"` | 1 | 1 | 1 | 0 |
| `"ab"` | 2 | 2 | 2 | 0 |
| `"abc"` | 3 | 3 | 3 | 0 |
| `"abcd"` | 4 | 4 | 4 | 0 |
| `"abcde"` | 5 | 5 | 5 | 0 |

_(Example: counting vowels in a string with no vowels — every character checked, none matched.)_

The count table is:
- **Manually fillable** — learners can trace code and fill it in by hand
- **Auto-verifiable** — the tracer tool can generate counts for comparison
- **The raw data** — every subsequent representation (ratio table, growth curve, Big O) is derived from this

## Manual Counting vs. Tracer

Learners do as much or as little manual counting as they need. The workflow:

1. **Trace by hand** for small inputs (n=1, 2, 3) — builds understanding of _what_ is being counted
2. **Predict** counts for n=4, 5 — tests whether the pattern is understood
3. **Run the tracer** to verify predictions and fill in larger input sizes
4. **Compare** manual counts with tracer output — discrepancies reveal misunderstandings

The tracer is a tool, not a replacement for understanding. Manual counting first ensures learners know what they're counting before automating it.

## Applied to Simple Algorithms

6.0 applies the counting methodology to the simplest algorithms from Chapter 5:

### Straight-line Code (O(1))

```js
const getFirstChar = (text) => {
  return text[0];
};
```

No loops. The count table has the same values for every input size. The pattern is trivially constant.

### Single-pass, No Decisions (O(n))

```js
const toUpperCase = (text) => {
  let result = '';
  for (const char of text) {
    result = result + char.toUpperCase();
  }
  return result;
};
```

One iteration per character. The count table shows iterations = n for every input. The pattern is obviously linear.

### Single-pass with Filter (O(n), from 5.1)

```js
const countVowels = (text) => {
  let count = 0;
  for (const char of text) {
    if ('aeiou'.includes(char.toLowerCase())) {
      count = count + 1;
    }
  }
  return count;
};
```

Iterations always = n. But assignments depend on input content (how many vowels). This introduces the idea that some counts depend on input SIZE and others on input CONTENT — a distinction that becomes important later.

## The Predict-Count-Compare Cycle

This is Chapter 2's predict-execute-compare cycle, adapted:

1. **Predict** — how many iterations/comparisons/assignments for this input?
2. **Count** — trace the code (manually or with tracer) and record actual counts
3. **Compare** — does the count match the prediction? If not, what was misunderstood?

The object of prediction shifts from "what happens next" (Ch2) to "how many times does this happen" (Ch6). Same cycle, different focus.

## Skills and Objectives

- 🥚 Counting the number of iterations, comparisons, and assignments a function executes for a given input
- 🥚 Filling in a count table for a simple function across increasing input sizes
- 🥚 Using the tracer tool to verify manual counts
- 🐣 Distinguishing between counts that depend on input SIZE (iterations in a for loop) and counts that depend on input CONTENT (how many vowels)
- 🐣 Predicting counts for a new input size based on the pattern observed in the count table, then verifying
- 🐣 Connecting the predict-count-compare cycle to predictive stepping from Chapter 2: same skill, different object of prediction
- 🐥 Choosing which operations to count based on what you want to learn about the algorithm — iterations for structure, comparisons for decisions, assignments for state changes
