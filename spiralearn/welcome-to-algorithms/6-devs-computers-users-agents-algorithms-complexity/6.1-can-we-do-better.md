# 6.1: Can We Do Better?

Different decompositions of the same problem produce different amounts of work. Learners have felt this qualitatively throughout Chapter 5. Now, with counting tools from 6.0 and named functions from 5.6–5.8, they can MEASURE it.

This subchapter is the motivating question for all of Chapter 6. It doesn't answer everything — it creates the need for formal tools (growth rate reasoning, Big O notation) that later subchapters will provide.

## The Setup

From Chapter 5, learners have three palindrome functions that satisfy the same JSDoc contract and pass the same unit tests:

```js
/**
 * Check if a string is a palindrome.
 * @param {string} text - The string to check.
 * @returns {boolean} True if the string reads the same forwards and backwards.
 */
```

Three implementations:
- `isPalindromeReverse` (5.1-style) — build the reversed string, compare
- `isPalindromeNested` (5.4-style) — nested loop comparing each pair of positions
- `isPalindromeConverging` (5.5-style) — two pointers from both ends

Same contract. Same tests. Same results. Different strategies. Different amounts of work.

## The Counting Exercise

Apply the count table from 6.0 to all three implementations:

| Input Size (n) | Reverse: iterations | Nested: iterations | Converging: iterations |

| 1              | 1                   | 0                  | 0                      |

| 5              | 5                   | 10                 | 2                      |

| 10             | 10                  | 45                 | 5                      |
| 20             | 20                  | 190                | 10                     |

_(Counts are approximate — exact values depend on implementation details. The pattern is what matters.)_

The qualitative feeling from Chapter 5 — "the nested version does way more work" — is now a measurable fact. The count table makes it visible.

## Introducing the Analytical Chain

This is the first substantive application of all 5 analysis questions:

1. **How many times does each operation execute?** → the count table above
2. **How do the counts change as input grows?** → Reverse grows proportionally (×2 when input ×2). Nested grows much faster (×4 when input ×2). Converging grows proportionally but at half the rate.
3. **Which counts matter?** → At n=20, the difference between 20 and 190 iterations is dramatic. At n=100, it would be 100 vs. ~5000. The nested count dominates.
4. **What causes this growth?** → Reverse and converging each scan once (5.1/5.5 patterns). Nested scans from each position to all others (5.4 pattern). The iteration structure — single-pass vs. nested — determines the growth shape.
5. **Which complexity class?** → We don't have the formal vocabulary yet. But we can observe: reverse and converging grow "like n." Nested grows "like n×n." These are different families.

And the separate optimization question:

6. **Could a different decomposition change the growth?** → Yes! That's the whole point. The converging decomposition does fundamentally less work than the nested one. Different strategy → different growth → different efficiency.

## The Qualitative Gradient

The internal progression mirrors the original "Can We Do Better?" design:

1. **Observe** — Run all three palindrome functions with the same inputs. Fill in the count table. See the numbers.
2. **Articulate** — Explain WHY the nested version does more work. Point to the iteration structure from Chapter 5: nested loops mean each position spawns a scan of other positions.
3. **Measure** — Use the count table to quantify HOW MUCH more work. At n=20, nested does ~10× more iterations than reverse. At n=100? The ratio keeps growing.
4. **"We need to formalize this"** — The count table shows the difference. But describing it precisely requires vocabulary we don't have yet. "Grows like n" vs. "grows like n×n" is close, but imprecise. This motivates the formal tools in subsequent subchapters.

## The First Pass Through the Representation Sequence

This subchapter introduces the first three representations from the chapter's sequence:

1. **Count table** — filled in for all three palindrome functions (done above)
2. **Ratio table** — what happens when input doubles? Reverse: iterations ×2. Nested: iterations ×4. Converging: iterations ×2.
3. **Growth curve** — sketch the three functions on the same axes. Two are straight lines (different slopes). One curves upward. The visual makes the difference unmistakable.

The remaining representations (growth formula, dominant term analysis, Big O) require tools this subchapter motivates but doesn't yet provide.

## Skills and Objectives

- 🥚 Filling in count tables for multiple implementations of the same function and comparing the counts
- 🥚 Identifying from count tables which implementation produces more work, and by how much, at specific input sizes
- 🐣 Applying the first 3 questions of the analytical chain (count, growth, dominance) to a concrete comparison
- 🐣 Connecting the growth difference to Chapter 5's iteration structure: single-pass algorithms grow differently from nested algorithms
- 🐣 Constructing a ratio table (what happens when input doubles?) and recognizing ×2 vs. ×4 growth as fundamentally different
- 🐥 Articulating why "Can we do better?" is now a formally answerable question — the count table replaces intuition with measurement
- 🐥 Recognizing that the need for precise vocabulary ("grows like n" vs. "grows like n×n") motivates formal notation (Big O) — creating the question that later subchapters answer

**Why this deserves its own subchapter:** "Can we do better?" is the transition from understanding algorithms to measuring them. It creates the motivation for everything that follows in Chapter 6. The counting tools from 6.0 are here deployed for the first time on a genuinely interesting comparison — and the limits of informal language ("grows like n×n") create the need for formal tools.
